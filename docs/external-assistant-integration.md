# External Assistant Integration

This document describes the HTTP contract for integrating an external assistant service with Chatwoot.

In this mode, Chatwoot does not call an LLM provider directly for Assistant, Copilot, or AI editor actions. Chatwoot sends a signed HTTP request to the configured external assistant service. The external service owns model selection, prompts, tools, memory, retrieval, and any long-term storage it needs.

## Request Flow

```text
Chatwoot frontend -> Chatwoot Rails backend -> External Assistant service
External Assistant service -> Chatwoot Rails backend -> Chatwoot frontend
```

Chatwoot stores only the external assistant connection settings:

- enabled/disabled
- service URL
- access token
- assistant ID
- context sharing flags

Copilot thread history, prompts, model settings, memory, and tool configuration can be stored by the external assistant service.

## Authentication

If an access token is configured in Chatwoot, each request includes:

```http
Authorization: Bearer <token>
Content-Type: application/json
Accept: application/json
```

The external service should treat this token as a shared secret and should not log it.

## Endpoint

Chatwoot sends a `POST` request to the configured `service_url`.

Example:

```http
POST https://assistant.example.com/chatwoot/assistant
```

### Top-Level Fields

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `assistant_id` | string, null | No | Assistant identifier configured in Chatwoot. The external service can use this to select prompt/model/tool settings. |
| `feature_name` | string, null | No | The Chatwoot feature that triggered the request. Usually present for AI editor tasks and label suggestions. Usually absent for Copilot chat. |
| `message` | string, null | No | The latest user/task input. |
| `thread_id` | string/integer, null | No | Copilot thread identifier. Chatwoot may generate a synthetic ID for stateless external mode. |
| `history` | array | No | Prior Copilot messages for the current thread. |
| `messages` | array | No | Prompt-style messages generated by Chatwoot for AI editor tasks. |
| `account` | object | Yes | Chatwoot account metadata. |
| `user` | object, null | No | Current Chatwoot user/agent. Can be null in background contexts. |
| `conversation` | object, null | No | Conversation context if available and enabled. |
| `settings` | object | Yes | Context sharing settings configured in Chatwoot. |

## Feature Names

The external service should be prepared for these `feature_name` values:

| Feature | Description |
| --- | --- |
| `reply_suggestion` | Suggest a reply for the current conversation. |
| `summarize` | Summarize the current conversation. |
| `follow_up` | Refine a previous AI editor result. |
| `fix_spelling_grammar` | Fix spelling and grammar in a draft message. |
| `improve` | Improve a draft message. |
| `casual` | Rewrite a draft message in a casual tone. |
| `professional` | Rewrite a draft message in a professional tone. |
| `friendly` | Rewrite a draft message in a friendly tone. |
| `confident` | Rewrite a draft message in a confident tone. |
| `straightforward` | Rewrite a draft message in a straightforward tone. |
| `label_suggestion` | Suggest labels for a conversation. |

Copilot chat requests may have `feature_name: null`. In that case, use `thread_id`, `history`, and `message` to handle the request as a chat turn.

## Response Body

The minimal successful response is:

```json
{
  "content": "Assistant response text"
}
```

Chatwoot also accepts `answer` or `message` as aliases for `content`:

```json
{
  "answer": "Assistant response text"
}
```

```json
{
  "message": "Assistant response text"
}
```

For Copilot, the external service may return a thread ID:

```json
{
  "content": "Assistant response text",
  "thread_id": "external-thread-123"
}
```

Chatwoot currently reads:

- `content`
- `answer`
- `message`
- `thread_id`
- `threadId`

## Error Responses

Any non-2xx response is treated as an error. Chatwoot uses the response body as the error message.

Recommended format:

```json
{
  "error": "Assistant is temporarily unavailable"
}
```

Recommended status codes:

| Status | Meaning |
| --- | --- |
| `400` | Invalid request body. |
| `401` | Missing or invalid bearer token. |
| `403` | Token is valid but not allowed for this account or assistant. |
| `422` | The request is valid but cannot be processed. |
| `429` | Rate limit exceeded. |
| `500` | Unexpected assistant service error. |
| `503` | Assistant service temporarily unavailable. |

## Minimal Node.js Example

```js
import express from 'express';

const app = express();
app.use(express.json({ limit: '2mb' }));

app.post('/chatwoot/assistant', async (req, res) => {
  const authHeader = req.header('authorization') || '';
  if (authHeader !== `Bearer ${process.env.CHATWOOT_ASSISTANT_TOKEN}`) {
    return res.status(401).json({ error: 'Unauthorized' });
  }

  const { assistant_id, feature_name, message, conversation, history } = req.body;

  // Select prompts, tools, model, and memory based on assistant_id/feature_name.
  // Call your LLM or agent runtime here.
  const content = [
    `Assistant: ${assistant_id || 'default'}`,
    `Feature: ${feature_name || 'copilot'}`,
    `Message: ${message || ''}`,
    `Conversation: ${conversation?.display_id || 'none'}`,
    `History turns: ${history?.length || 0}`
  ].join('\n');

  return res.json({
    content,
    thread_id: req.body.thread_id
  });
});

app.listen(3000, () => {
  console.log('External assistant listening on http://localhost:3000');
});
```

## Implementation Notes

- Treat `messages` as the highest-fidelity prompt context for AI editor tasks.
- Treat `history` as the Copilot chat history for the current thread.
- `conversation.messages` includes up to the latest conversation messages sent by Chatwoot, depending on the configured context flags.
- Private notes are sent only when `settings.send_private_notes` is `true`.
- Contact details are sent only when `settings.send_contact_details` is `true`.
- Conversation messages are sent only when `settings.send_conversation_context` is `true`.
- The service should be idempotent enough to tolerate client retries.
- The service should avoid logging PII unless explicitly required and protected.
